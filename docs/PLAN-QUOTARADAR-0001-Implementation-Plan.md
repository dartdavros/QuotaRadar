# PLAN-QUOTARADAR-0001 — План реализации QuotaRadar

- **Статус:** утверждён; этапы 1–6 реализованы локально
- **Версия:** 1.1
- **Дата:** 2026-07-20
- **Связанные документы:**
  - `ARCH-QUOTARADAR-0001-System-Architecture.md`
  - `SPEC-QUOTARADAR-0001-Technical-Specification.md`
  - `ADR-QUOTARADAR-0002-Recovery-of-Orphaned-Work.md`

## 1. Цель плана

Реализовать QuotaRadar в согласованной архитектуре без расширения функциональных границ:

- мониторинг `@OpenAIDevs` и `@ClaudeDevs` через официальный X API v2;
- ИИ-интерпретация публикаций;
- автоматическое формирование русскоязычных уведомлений;
- доставка в Telegram-каналы и личные чаты;
- управление через Django Admin;
- обязательный прокси для X, ИИ-провайдера и Telegram;
- зашифрованное хранение секретов в PostgreSQL;
- развёртывание через Docker Compose.

Ручная модерация, RSS, scraping X, другие источники, публичный веб-интерфейс, платные функции и пользовательские фильтры не входят в реализацию.

## 2. Принципы выполнения

1. Этапы выполняются последовательно.
2. Каждый этап имеет самостоятельный проверяемый результат.
3. Следующий этап начинается после прохождения критериев приёмки предыдущего.
4. Архитектурные отклонения от ARCH/SPEC не вносятся скрыто.
5. Необходимое отклонение сначала оформляется отдельным решением и утверждается.
6. Все внешние HTTP/HTTPS-запросы выполняются только через общий прокси-клиент.
7. Секреты не передаются в аргументах Celery-задач и не попадают в логи.
8. Повторный запуск задач не должен создавать повторные анализы и доставки.

# Этап 1. Базовый проект и Docker-контур

## Цель

Создать запускаемый каркас QuotaRadar с Django, PostgreSQL, Redis, Celery и Telegram-процессом в едином Docker Compose-контуре.

## Работы

### 1.1. Структура проекта

Создать Django-проект и приложения:

```text
configuration
secrets
sources
monitoring
analysis
telegram
```

### 1.2. Зависимости

Подключить минимально необходимые зависимости:

- Django;
- PostgreSQL driver;
- Celery;
- Redis client;
- HTTP-клиент с поддержкой HTTP/HTTPS proxy;
- Telegram Bot API library или собственный клиент;
- `cryptography`;
- библиотеку валидации structured output;
- production WSGI/ASGI server.

### 1.3. Docker

Создать:

- один `Dockerfile` для прикладных процессов;
- `docker-compose.yml`;
- сервисы `init`, `web`, `bot`, `worker`, `beat`, `postgres`, `redis`;
- профиль `tools` с отдельным интеграционным сервисом `test`;
- persistent volume PostgreSQL;
- healthcheck PostgreSQL и Redis;
- read-only mount для master key;
- команды запуска каждого процесса.

### 1.4. Bootstrap-конфигурация

Предусмотреть только параметры, необходимые до чтения PostgreSQL:

```text
DJANGO_SECRET_KEY
DJANGO_ALLOWED_HOSTS
QUOTARADAR_WEB_BIND_ADDRESS
POSTGRES_DB
POSTGRES_USER
POSTGRES_PASSWORD
```

Внутри Docker Compose фиксируются bootstrap-адреса `POSTGRES_HOST`, `POSTGRES_PORT`, `REDIS_URL` и `QUOTARADAR_MASTER_KEY_FILE`. Для совместимости standalone-конфигурация Django также принимает `DATABASE_URL` с приоритетом над дискретными PostgreSQL-параметрами.

Создание superuser выполняется отдельной воспроизводимой командой.

### 1.5. Базовые проверки

Добавить:

- Django system check;
- проверку доступности PostgreSQL;
- проверку доступности Redis;
- проверку чтения master key;
- базовую конфигурацию тестов.

## Результат этапа

Все контейнеры запускаются, Django Admin открывается, миграции выполняются, Celery Worker и Celery Beat подключаются к Redis, процесс бота запускается без внешних запросов.

## Критерии приёмки

1. Проект собирается одним Dockerfile.
2. `docker compose up` запускает production-контур `init`, `web`, `bot`, `worker`, `beat`, `postgres`, `redis`.
3. PostgreSQL и Redis используют внутреннюю Docker-сеть.
4. Django Admin доступен после создания superuser.
5. Worker принимает тестовую Celery-задачу.
6. Beat запускается без ошибок конфигурации.
7. Master key доступен только прикладным контейнерам и смонтирован read-only.
8. В репозитории отсутствуют реальные секреты.

# Этап 2. Конфигурация, секреты и общий прокси-клиент

## Цель

Реализовать централизованное управление настройками и секретами через Django Admin, а также единственный разрешённый путь выполнения внешних запросов.

## Работы

### 2.1. Модели конфигурации

Реализовать:

- `SystemConfiguration`;
- `PromptTemplate`;
- ограничение на одну активную системную конфигурацию;
- начальную версию промпта.

### 2.2. Зашифрованные секреты

Реализовать `EncryptedSecret` для кодов:

```text
telegram_bot_token
llm_api_key
x_bearer_token
proxy_url
```

Требования:

- аутентифицированное симметричное шифрование;
- ciphertext в PostgreSQL;
- master key вне PostgreSQL;
- `key_version`;
- сервисный слой чтения и записи;
- запрет прямого использования поля `encrypted_value` интеграциями.

### 2.3. Django Admin

Реализовать:

- просмотр расшифрованных значений;
- изменение секретов;
- отдельные permissions на просмотр и изменение;
- аудит `updated_at` и `updated_by`;
- отсутствие секретов в списках, URL и Django messages.

### 2.4. Общий HTTP client factory

Реализовать централизованный factory, который:

- получает расшифрованный `proxy_url` через secret service;
- создаёт HTTP-клиент с обязательным proxy;
- применяет общие timeout-настройки;
- маскирует proxy URL в исключениях и логах;
- запрещает прямой внешний запрос без настроенного proxy.

Интеграции X, ИИ и Telegram не должны самостоятельно создавать независимые HTTP-клиенты.

### 2.5. Диагностика

Добавить безопасные проверки конфигурации:

- секрет настроен или отсутствует;
- master key корректен;
- proxy URL синтаксически валиден;
- тестовый внешний запрос использует proxy;
- тело ошибки не содержит credentials.

## Результат этапа

Администратор полностью управляет рабочими параметрами и секретами через Django Admin. Любая будущая внешняя интеграция может работать только через общий прокси-клиент.

## Критерии приёмки

1. В PostgreSQL отсутствуют открытые значения секретов.
2. Уполномоченный superuser видит расшифрованные секреты в Django Admin.
3. Пользователь без permission не видит секреты.
4. Изменённый секрет корректно шифруется повторно.
5. Неверный master key не приводит к показу повреждённого значения.
6. Без `proxy_url` внешний запрос завершается конфигурационной ошибкой.
7. Proxy URL, токены и API-ключи отсутствуют в логах и traceback-тексте.
8. Unit-тесты покрывают шифрование, расшифровку, permissions и HTTP client factory.

# Этап 3. Интеграция с X API и сохранение постов

## Цель

Надёжно и идемпотентно получать новые публикации `@OpenAIDevs` и `@ClaudeDevs` через официальный X API v2.

## Работы

### 3.1. Модели данных

Реализовать:

- `Source`;
- `SourcePost`;
- начальные записи `OpenAIDevs` и `ClaudeDevs`;
- уникальность `SourcePost.external_id`;
- статусы обработки поста.

### 3.2. X API client

Реализовать вызовы через общий прокси-клиент:

```http
GET /2/users/by?usernames=OpenAIDevs,ClaudeDevs
GET /2/users/{user_id}/tweets
```

Поддержать:

- App-Only Bearer Token;
- первичную загрузку одной страницы с настраиваемым числом последних постов, если у источника нет курсора и сохранённых постов;
- `since_id`;
- настраиваемый размер страницы для последующего polling;
- пагинацию `next_token`;
- `exclude=retweets`;
- replies без исключения;
- необходимые поля и expansions из SPEC;
- обработку длинного текста, цитат, ссылок и alt text.

### 3.3. Celery polling

Реализовать задачи:

```text
poll_sources
poll_source(source_id)
```

Требования:

- запуск через Celery Beat;
- интервал из `SystemConfiguration`;
- Redis-lock на источник;
- сохранение постов от старого к новому;
- обновление `last_post_id` только после успешного сохранения страницы данных;
- повторный запуск не создаёт дубликаты.

### 3.4. Ошибки X API

Реализовать поведение для:

- `401`;
- `403`;
- `404`;
- `429` с `x-rate-limit-reset`;
- `5xx` и сетевых ошибок;
- недоступного прокси.

### 3.5. Админка и диагностика

В Django Admin отображать:

- состояние источника;
- X User ID;
- последний Post ID;
- время последней проверки;
- время последнего успеха;
- последнюю ошибку;
- сохранённые посты и raw response.

## Результат этапа

QuotaRadar автоматически получает новые посты двух источников через прокси и сохраняет их в PostgreSQL без повторов. ИИ и Telegram пока не используются.

## Критерии приёмки

1. Оба X User ID разрешаются и сохраняются.
2. Все обращения к `api.x.com` подтверждённо проходят через proxy.
3. Репосты не сохраняются.
4. Replies сохраняются.
5. Новый пост создаётся один раз.
6. Повторный polling не создаёт дубликат.
7. Пагинация не пропускает посты.
8. После временной ошибки следующий успешный polling продолжает работу с корректного `since_id`.
9. `429` обрабатывается по времени reset.
10. Интеграционные тесты используют сохранённые fixtures ответов X API, а не реальные платные запросы.

# Этап 4. ИИ-интерпретация и формирование русского сообщения

## Цель

Автоматически определить релевантность каждого нового поста и сформировать точное русскоязычное уведомление.

## Работы

### 4.1. Модель анализа

Реализовать `Analysis` с one-to-one связью с `SourcePost`.

Хранить:

- релевантность;
- тип события;
- provider;
- product;
- заголовок и сообщение на русском;
- модель;
- версию промпта;
- raw response;
- ошибку анализа.

### 4.2. Нормализация контента

Сформировать единый текстовый контекст из:

- основного текста;
- `note_tweet`;
- article text;
- процитированного поста;
- раскрытых URL;
- alt text.

Изображения и видео не анализируются в версии 1.0.

### 4.3. LLM adapter

Реализовать адаптер ИИ-провайдера, использующий:

- `llm_base_url`;
- `llm_model`;
- параметры генерации;
- активный `PromptTemplate`;
- `llm_api_key`;
- общий proxy HTTP client.

Адаптер не должен быть связан с конкретным поставщиком за пределами конфигурации endpoint и формата API, принятого в SPEC.

### 4.4. Structured output

Реализовать серверную схему:

```text
is_relevant
event_type
provider
product
title_ru
message_ru
```

Допустимые события:

```text
quota_reset
quota_increase
quota_extension
```

### 4.5. Celery-задача анализа

Реализовать:

```text
analyze_post(source_post_id)
```

Требования:

- один `Analysis` на один `SourcePost`;
- retry для timeout, `5xx` и невалидного structured output;
- отсутствие публикации при окончательной ошибке;
- нерелевантный пост получает конечный статус без дальнейшей доставки.

### 4.6. Контроль качества сообщения

Приложение проверяет:

- наличие русского заголовка и текста для релевантного события;
- соответствие provider/product источнику;
- допустимый тип события;
- отсутствие пустого сообщения;
- максимальную длину Telegram-сообщения;
- добавление source URL самим приложением, а не моделью.

## Результат этапа

Каждый новый пост X автоматически получает валидированный результат анализа. Релевантные посты имеют готовый русский текст, нерелевантные завершают обработку без отправки.

## Критерии приёмки

1. Все запросы к ИИ-провайдеру проходят через proxy.
2. Ответ не по схеме не сохраняется как успешный анализ.
3. Один пост не получает несколько успешных анализов.
4. `quota_reset`, `quota_increase`, `quota_extension` корректно проходят классификацию на fixtures.
5. Нерелевантные публикации не переходят к доставке.
6. Русский текст не содержит добавленной моделью неподтверждённой ссылки.
7. Source URL добавляется приложением.
8. В записи анализа сохраняются модель и версия промпта.
9. Unit-тесты покрывают валидные, невалидные и нерелевантные ответы.

# Этап 5. Telegram-бот, канал и доставка

## Цель

Реализовать автоматическую доставку релевантных сообщений в Telegram-каналы и личные чаты.

## Работы

### 5.1. Модели доставки

Реализовать:

- `DeliveryTarget`;
- `Delivery`;
- типы `channel` и `private_chat`;
- уникальность `analysis + target`;
- статусы `pending`, `sent`, `failed`.

### 5.2. Telegram API client

Реализовать через общий proxy HTTP client:

- `getUpdates`;
- `sendMessage`;
- обработку Telegram API errors;
- безопасное использование `telegram_bot_token`.

### 5.3. Процесс бота

Реализовать отдельный контейнер `bot` на основе Django management command.

Команды:

```text
/start
/stop
/status
```

Поведение:

- `/start` создаёт или активирует `private_chat` target;
- `/stop` отключает target;
- `/status` показывает состояние подписки;
- повторная команда не создаёт дубликаты.

### 5.4. Каналы

Через Django Admin обеспечить:

- создание `channel` target;
- включение и отключение;
- указание Telegram chat ID или `@username` канала;

### 5.5. Доставка

Реализовать задачу:

```text
deliver_analysis(analysis_id, target_id)
```

Формат:

```text
{title_ru}

{message_ru}

Источник: {source_url}
```

Требования:

- отдельная задача на каждого получателя;
- ошибка одного target не блокирует остальные;
- успешная доставка не повторяется;
- временные ошибки повторяются;
- постоянная ошибка личного чата отключает target согласно SPEC;
- ошибка канала сохраняется без автоматического отключения.

## Результат этапа

QuotaRadar выполняет полный автоматический цикл: X → ИИ → русское сообщение → Telegram-канал и/или личный чат.

## Критерии приёмки

1. Все обращения к Telegram Bot API проходят через proxy.
2. `/start`, `/stop`, `/status` работают идемпотентно.
3. Релевантный анализ отправляется в настроенный канал.
4. Подписанный пользователь получает личное сообщение.
5. Отключённый пользователь сообщение не получает.
6. Сообщение содержит исходную ссылку X.
7. Один анализ не отправляется одному target повторно.
8. Ошибка одного target не прерывает доставку остальным.
9. Telegram token и proxy URL отсутствуют в логах.

# Этап 6. Стабилизация и open-source выпуск

- **Статус реализации:** выполнен локально; чистая серверная приёмка ожидает целевое окружение.

## Цель

Подготовить QuotaRadar к воспроизводимому самостоятельному развёртыванию и публикации на GitHub.

## Работы

### 6.1. Сквозные тесты

Покрыть сценарии:

- новый нерелевантный пост;
- сброс квоты;
- повышение квоты;
- продление повышенной квоты;
- повторное получение поста и отсутствие нового `Analysis`/`Delivery`;
- аварийное окно между сохранением релевантного `Analysis` и созданием `Delivery`;
- временная ошибка X;
- ошибка ИИ;
- невалидный structured output;
- временная и постоянная ошибка Telegram;
- отсутствие proxy;
- неверный master key.

Внешние API в автоматических тестах заменяются fixtures/mocks.

### 6.2. Безопасность

Проверить:

- ciphertext в дампе PostgreSQL;
- permissions Django Admin;
- отсутствие секретов в логах;
- отсутствие секретов в Celery task arguments;
- read-only master key mount;
- архитектурные guard-тесты на прямое создание внешнего HTTP-клиента;
- невозможность прямого внешнего запроса в обход proxy factory.

### 6.3. Наблюдаемость

Зафиксировать структурированные логи с идентификаторами:

- task;
- source;
- X Post;
- analysis;
- delivery target;
- delivery;
- status и error type.

Не логировать тела запросов и ответов, содержащие секреты.

### 6.4. Развёртывание

Подготовить:

- `.env.example` только с bootstrap-параметрами;
- генерацию master key;
- one-shot `init` для checks, миграций и static files;
- создание superuser;
- ввод секретов через Django Admin;
- создание Telegram-канала target;
- команды Docker Compose для запуска, обновления, логов и остановки;
- команды для PowerShell.

### 6.5. GitHub

Подготовить:

- `README.md`;
- лицензию;
- описание архитектуры;
- инструкцию X Developer Console;
- инструкцию создания Telegram-бота;
- настройку обязательного proxy;
- инструкцию self-hosted режима личных уведомлений;
- security notice о master key и секретах;
- contribution guide;
- release notes версии `0.1.0`;
- GitHub Actions CI с реальными PostgreSQL и Redis.

### 6.6. Финальная приёмка

Выполнить чистое развёртывание на новом окружении только по документации репозитория и проверить полный сценарий на тестовой публикации/fixture без ручного изменения базы.

## Результат этапа

Воспроизводимый open-source релиз QuotaRadar, готовый к публикации на GitHub и дальнейшей подготовке статьи на Хабре.

## Критерии приёмки

1. Чистая установка выполняется по README без недокументированных действий.
2. PostgreSQL и Redis проходят healthcheck, прикладные процессы запускаются и остаются работоспособными.
3. Полный сценарий X → ИИ → Telegram проходит автоматически.
4. Все внешние запросы проходят через proxy.
5. Повторный запуск не создаёт новую запись доставки и не переотправляет подтверждённо успешную доставку; ограничение неопределённого результата Telegram зафиксировано в ADR-0002.
6. Реальные секреты отсутствуют в Git-истории и артефактах.
7. Автоматические тесты проходят в Docker-контуре.
8. ARCH, SPEC и PLAN соответствуют фактической реализации.
9. Механизм восстановления потерянных задач и граница Telegram exactly-once оформлены в `ADR-QUOTARADAR-0002`.
10. Чистое серверное развёртывание остаётся обязательной внешней приёмкой после публикации на целевом хосте.

## 3. Последовательность зависимостей

```text
Этап 1: Docker и базовый проект
            ↓
Этап 2: конфигурация, секреты и proxy
            ↓
Этап 3: получение постов X
            ↓
Этап 4: ИИ-анализ
            ↓
Этап 5: Telegram-доставка
            ↓
Этап 6: стабилизация и open-source выпуск
```

Параллельная реализация этапов 3–5 не рекомендуется: каждый из них использует контракты и инфраструктуру предыдущего этапа.

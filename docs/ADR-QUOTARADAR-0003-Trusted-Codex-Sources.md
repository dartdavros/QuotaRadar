# ADR-QUOTARADAR-0003 — Доверенные источники событий Codex

- **Статус:** принято
- **Дата:** 2026-07-21
- **Связанные документы:**

  - `SPEC-QUOTARADAR-0001-Technical-Specification.md`
  - `PLAN-QUOTARADAR-0001-Implementation-Plan.md`

## 1. Контекст

Первоначальная версия QuotaRadar использовала `@OpenAIDevs` как единственный X-источник событий Codex. Практическая эксплуатация показала, что массовые сбросы и временные изменения квот Codex могут объявляться в личных аккаунтах руководителей OpenAI и не публиковаться в `@OpenAIDevs`.

Отказ от X API, добавление scraping, RSS или неавторитетных агрегаторов не требуется. Необходимо расширить контролируемый allowlist X-аккаунтов и сохранить существующую модель доверия, дедупликации и ИИ-классификации.

## 2. Решение

### 2.1. Allowlist источников

Для Codex используются следующие доверенные X-аккаунты:

- `@thsottiaux` — основной дополнительный источник объявлений о квотах Codex;
- `@OpenAIDevs` — официальный продуктовый источник OpenAI;
- `@sama` — резервный источник массовых объявлений OpenAI.

Для Claude Code сохраняется:

- `@ClaudeDevs` — официальный продуктовый источник Anthropic.

Каждый источник хранится отдельной записью `Source`. Привязка к продукту выполняется через `provider`: все источники `openai` анализируются как Codex, источник `anthropic` — как Claude Code.

### 2.2. Управление источниками

Список доверенных аккаунтов создаётся миграциями и не расширяется вручную через Django Admin. В Admin разрешено только включать и отключать уже утверждённые записи.

Новые источники `@thsottiaux` и `@sama` создаются выключенными. Это предотвращает автоматический первый polling сразу после production-миграции до того, как оператор проверит `bootstrap_post_limit` и решит, какой объём истории загружать.

Оператор должен:

1. оставить общий мониторинг выключенным либо временно выключить его;
2. установить требуемый `bootstrap_post_limit`;
3. включить `@thsottiaux`;
4. при необходимости отдельно включить резервный `@sama`;
5. включить общий мониторинг.

### 2.3. Политика ИИ-классификации

Создаётся `quota_event_classifier v2`.

Промпт учитывает, что доверенный источник может быть корпоративным или персональным аккаунтом. Релевантным считается только прямое фактическое объявление о массовом изменении квот для группы пользователей, тарифа или всех пользователей.

Предположения, обсуждения, статистика использования, ответы без нового объявления, общие продуктовые новости и изменения одного аккаунта остаются нерелевантными.

При миграции системная конфигурация автоматически переключается с стандартного `quota_event_classifier v1` на `v2`. Пользовательский активный промпт с другим кодом или версией не переопределяется.

## 3. Последствия

Положительные:

- QuotaRadar получает объявления Codex, отсутствующие в `@OpenAIDevs`;
- сохраняются официальный X API, существующий polling и единый pipeline;
- новые источники не начинают polling до явного действия оператора;
- резервный источник можно включить без изменения кода;
- более строгий промпт снижает риск публикации нерелевантных постов персональных аккаунтов.

Отрицательные:

- включение дополнительных аккаунтов увеличивает количество X reads и ИИ-анализов;
- персональные аккаунты публикуют больше нерелевантного контента;
- полнота мониторинга зависит от актуальности утверждённого allowlist;
- добавление следующего источника по-прежнему требует архитектурного решения и миграции.

## 4. Отклонённые варианты

### Использовать только `@OpenAIDevs`

Отклонено, потому что этот аккаунт не покрывает все массовые события квот Codex.

### Включить `@sama` автоматически

Отклонено из-за высокого объёма общего контента и дополнительных затрат на ИИ-классификацию. Источник оставлен резервным и включается оператором.

### Разрешить произвольное добавление источников через Django Admin

Отклонено, потому что это разрушает контролируемый allowlist и позволяет без ревью направить произвольный контент в автоматический pipeline публикации.

### Добавить scraping, RSS или агрегаторы

Отклонено: текущий X API pipeline достаточен, а альтернативы добавляют новые риски доверия, устойчивости и поддержки.
